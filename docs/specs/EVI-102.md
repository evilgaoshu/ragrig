# EVI-102: Operational Console Source/Run Mutating Workflows P1

## 目标

把 Web Console 从只读 MVP 推进到安全操作台——支持 source 配置草稿、ingestion dry-run、失败检查、retry/resume，所有写操作可审计可回滚。

## Mutating API

### POST /sources/validate-config
- 请求：`{ plugin_id, config }`
- 校验 source 配置草稿：插件注册、secret 引用格式（env:VAR 必须）、依赖可用性
- 返回结构化状态：`valid`, `status`（ready/degraded/disabled）, `reason`, `missing_dependencies`, `credential_issues`
- 缺依赖或凭据时显示 degraded/disabled，不伪造可用状态
- 对 secret 关键字（api_key, password, token, secret, private_key, access_key, session_token）的原始值拒绝，必须使用 env: 引用

### POST /sources
- 请求：`{ plugin_id, config, knowledge_base, operator? }`
- 保存已验证的 source 配置到数据库（创建或更新 Source 记录）
- 支持 local/fileshare/S3 三种 source kind
- 返回保存后的 source 详情

### POST /sources/dry-run
- 请求：`{ plugin_id, config }`
- 执行扫描并返回候选文件列表、skip reason、failure reason
- **不写入** document_versions / chunks / embeddings
- 对 local 使用 `scan_paths`，S3 使用 `scan_objects`，fileshare 使用 `scan_files`
- 返回：`{ dry_run: true, total, discovered, skipped, failed }`

### GET /pipeline-run-items/{item_id}
- 返回单个 pipeline run item 详情，包括 config_snapshot

### POST /pipeline-runs/{run_id}/retry
- 请求：`{ operator?, new_snapshot? }`
- 重新处理 run 中所有 failed items
- `new_snapshot=false`（默认）：复用原 run 的 `config_snapshot_json`
- `new_snapshot=true`：创建新快照（预留，当前始终使用原 config snapshot）
- 返回 retry 摘要：`{ retried, succeeded, failed, items }`
- 不静默修改历史 run，对每个 item 创建新的 DocumentVersion

### POST /pipeline-run-items/{item_id}/retry
- 请求：`{ operator? }`
- 重新处理单个 failed item，复用原 config snapshot
- 返回 item 新状态：`{ status, new_version_number }`

## 表单状态

### Source Config Form
- **empty**：无 source 时显示空状态，展示配置表单入口
- **loading**：点击 Validate/Save/Dry-run 时显示加载状态
- **valid/ready**：验证成功、依赖和凭据齐全，显示绿色 pill
- **valid/degraded**：验证成功但缺依赖或凭据，显示黄色 pill 和原因
- **invalid/disabled**：配置错误、secret 原始值、plugin 未注册，显示红色 pill 和错误信息

### Pipeline Run Items
- **success**：绿色 pill
- **skipped**：黄色 pill，显示 skip reason
- **failed**：红色 pill，显示 error_message 和 Retry 按钮
- **retrying**：点击 retry 后更新状态

## Dry-run 语义

- 只读扫描：列出候选文件、skip reason、failure reason
- 不写入 document_versions / chunks / embeddings
- 不对实际 ingestion 产生副作用
- 对 local 类型直接扫描文件系统
- 对 S3/fileshare 类型创建客户端连接后扫描

## Retry/Resume 语义

- 每个 retry 复用了原 run 的 config_snapshot_json（不变的历史关联）
- 对每个 retry 的 item 创建新的 DocumentVersion（新版本号递增）
- 更新 item 状态为 success/failed 并设置新 error_message
- 更新 run 的 success_count/failure_count
- 不删除或修改历史 document_versions / chunks / embeddings
- 不对原 run 进行修改（状态字段除外）

## 失败队列

- 失败 item 通过 `retry_pipeline_run()` 批量处理或 `retry_pipeline_run_item()` 单独处理
- retry 后 item 重新 parse，失败原因保留在 error_message
- 无自动重试机制（需要用户点击触发）

## 审计事件

- Source 保存操作通过 `save_source_config()` 记录
- Retry 操作记录 operator 信息（通过 `x-operator` header）
- 所有写入操作不泄露 secret 原文

## Secret 边界

- 所有 secret 字段必须使用 `env:VAR_NAME` 引用
- `validate_source_config()` 对 raw secret 值返回 invalid，不执行保存
- `_find_raw_secret_values()` 递归检查嵌套 dict/list
- `_CONSOLE_FORBIDDEN_FRAGMENTS` 用于 retry error 脱敏
- 响应和日志中不包含 secret 明文

## 不做边界

- 不做完整账号权限系统
- 不接生产调度器
- 不自动运行破坏性删除
- 不把所有 connector 一次性做完

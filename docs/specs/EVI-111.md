# SPEC: EVI-111 Retrieval Benchmark Fixture ID Cross-Workspace Stability

**Issue**: EVI-111  
**Version**: 1.0  
**Date**: 2026-05-13  
**Status**: Implemented

---

## 1. Goal

Make `scripts.retrieval_benchmark_compare` and `scripts.retrieval_benchmark_baseline_refresh` derive the same `fixture_id` for the same corpus content regardless of checkout absolute path.

---

## 2. Stable Identity Boundary

`fixture_id` must be derived from the fixture corpus itself, not from the machine-local checkout path.

Included in the identity hash:

- relative file paths under the fixture root
- file bytes for every fixture file

Excluded from the identity hash:

- absolute checkout path
- temporary workspace root
- filesystem-specific inode / mtime metadata

This keeps `fixture_id` stable across workspaces while remaining sensitive to corpus edits, including content-only changes that do not change file size.

---

## 3. Compatibility Policy

Existing baseline manifests produced before EVI-111 may contain path-derived `fixture_id` values. Those manifests are treated as incompatible with the new content-derived computation.

Migration path:

1. Run `make retrieval-benchmark-baseline-refresh`
2. Commit the refreshed baseline JSON and manifest JSON
3. Re-run `make retrieval-benchmark-compare`

Compare failures caused by this condition must include an actionable refresh hint so operators can distinguish a real corpus change from a legacy baseline.

---

## 4. Tests

Required unit coverage:

1. Same fixture content under different temporary roots yields the same `fixture_id`
2. Content changes with unchanged file size yield a different `fixture_id`
3. Legacy baseline mismatch returns a migration/refresh message

---

## 5. Non-Goals

- No retrieval algorithm tuning
- No benchmark threshold changes
- No evidence pack structure redesign
- No external provider dependency

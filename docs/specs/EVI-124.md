# EVI-124: Async Task Queue for Long-Running Operations

## Goal

Move embedding generation, large-file parsing, and full index rebuild work out of
FastAPI request handling. API endpoints should return a task id immediately and
let clients poll task status instead of waiting for synchronous indexing or DAG
execution.

## Hard Requirements

1. `POST /knowledge-bases/{kb_name}/upload` returns after file intake with a
   `task_id` and no longer runs indexing synchronously.
2. Add `GET /tasks/{task_id}` returning `task_id`, `status`, `result`, and
   `error`, where status is one of `pending`, `running`, `completed`, or
   `failed`.
3. Add a `TaskRecord` database model/table with `id`, `status`, `task_type`,
   `payload_json`, `result_json`, `error`, `created_at`, and `updated_at`.
4. Add an Alembic migration that upgrades cleanly to head.
5. Provide an async execution layer through an in-process queue or thread pool
   with an interface that can later be replaced by Redis, ARQ, Celery, or
   another external queue.
6. `POST /pipeline-dags/ingestion` returns a `task_id` and executes the DAG in
   the background.
7. Concurrent upload requests must not block `/health`.

## Best-Effort Goals

- Progress push via WebSocket or SSE.
- One automatic retry after task failure.

## Out Of Scope

- Introducing Redis, Celery, ARQ, or another external queue dependency.
- Rewriting parser, chunker, or embedding provider internals.
- Migrating endpoints that are not long-running.

## Validation

- `uv run pytest tests/`
- `uv run alembic upgrade head --sql`
- GitHub PR checks for lint, tests, coverage, smoke jobs, and SQLite warnings.
